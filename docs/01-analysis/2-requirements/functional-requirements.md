# Функциональные требования

> Этап 2. Что система должна делать (функции, поведение).
> Приоритеты: **Must** — к запуску; **Should** — желательно; **Won't** — вне скоупа.

> **Скоуп — клиентское мобильное приложение и API для него** (в учебном проекте — mock).

## Границы скоупа

- **Слоты** приходят из API (mock). CRUD расписания — вне скоупа.
- **Программы и мастера** — read-only справочники.

## Запись на слот

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| FR-10 | Запись на выбранный слот. | Must | US-5 |
| FR-11 | Выбор «своё / прокат» для каждого места. | Must | US-6 |
| FR-12 | Одной записью 1–3 места (себя + до 2 гостей); без имён гостей (R-013). | Must | US-7 |
| FR-13 | `max_seats = min(free_seats, program.capacity_cap, 3)`; новичковая ≤6, на круге ≤10. | Must | US-8 |
| FR-14 | Прокат отдельно: `rental_count ≤ free_rental_kits`; «своё» не уменьшает прокатный фонд. | Must | US-6 |
| FR-15 | Запрет овербукинга и двойной брони. | Must | BR-1 |
| FR-22 | При создании брони клиент передаёт обязательный `Idempotency-Key`; повтор с тем же ключом и телом не создаёт дубль. | Must | R-022; US-14 |
| FR-45 | `price_total` из API — read-only; клиент не пересчитывает. | Must | R-005 |

> **Модель цены:** `price_total = price × seats_count + rental_price × rental_count` (RUB).
