# Use cases / сценарии

> Этап 2. Сценарии использования с **альтернативными потоками** и **исключениями**.
>
> **Скоуп: клиентское приложение и API для него.**

## UC-1: Запись на мастер-класс (себя и гостей)

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован; есть слот со свободными местами.
- **Триггер:** Клиент хочет забронировать место на занятии.

> **R-013:** По гостям — только количество мест и выбор проката. Имена не собираются.

### Основной поток
1. Клиент открывает список слотов (7 дней по умолчанию).
2. (Опц.) Применяет фильтры: дата, программа (лепка / гончарный круг), мастер, только свободные.
3. Открывает карточку слота (программа, мастер, цена, места, прокат).
4. Указывает число мест (1–3).
5. Для каждого места выбирает «своё» / «прокат».
6. Подтверждает запись (передаёт `Idempotency-Key`).
7. Система проверяет `max_seats` и `rental_count ≤ free_rental_kits`, создаёт запись.
8. Показывает подтверждение с ценой (`price_total`); запись в «Мои бронирования».

### Альтернативные потоки
- **A1. Все места — своё.** Прокатный фонд не затрагивается.
- **A2. Часть — прокат.** Уменьшается только `free_rental_kits` на число прокатных мест.
- **A3. Фильтр пуст.** Предложение сбросить/изменить фильтры.

### Исключения / ошибки
- **E1. Не хватает мест.** → Отказ, показать актуальное `free_seats`.
- **E2. Не хватает проката.** → Предложить уменьшить прокат или выбрать «своё».
- **E3. Гонка / слот заполнился.** → Отказ без овербукинга, обновить список.
- **E4. Сетевой сбой.** → Error/Retry; повтор с тем же `Idempotency-Key`.
- **E5. Слот отменён мастерской или начался.** → Отказ (`slot_cancelled` / `slot_started`).

### Матрица ошибок (ключевые операции)

| Код | UX-реакция |
| :-- | :-- |
| `slot_full` | Показать актуальные свободные места/прокат |
| `double_booking` | «Бронь на этот слот уже есть» → «Мои бронирования» |
| `slot_cancelled` | «Занятие отменено мастерской» + причина, CTA неактивна |
| `slot_started` | «Занятие уже началось», запись недоступна |
| 5xx / timeout | Error/Retry; повтор брони с тем же `Idempotency-Key` |

### Постусловия
- Успех: запись создана, счётчики уменьшены, запись в «Мои бронирования».
- Ошибка: состояние слота не изменилось.

## UC-2: Отмена записи

- **Актёры:** Клиент
- **Предусловия:** Есть предстоящая запись; занятие не началось.
- **Триггер:** Клиент решает отменить участие.

> **R-014:** Отмена только целиком — все места и прокат.

### Основной поток
1. Клиент открывает «Мои бронирования», выбирает запись.
2. Нажимает «Отменить».
3. Сервер определяет время до начала (UTC, отображение в локальной зоне).
4. **Ранняя (≥2 ч):** статус «отменена»; места и прокат возвращаются в слот.
5. Подтверждение отмены.

### Альтернативные потоки
- **A1. Поздняя (<2 ч).** Статус `late_cancel`; места не освобождаются; штрафов нет.
- **A2. Расхождение клиент/сервер.** Применяется серверный статус.

### Исключения
- **E1. Занятие началось.** Отмена недоступна.
- **E2. Уже отменена.** Показать текущий статус.
- **E3. Офлайн.** Мутация недоступна, понятное сообщение.

### Постусловия
- Статус обновлён; при ранней отмене места снова доступны.

## UC-3: Просмотр и фильтрация списка слотов

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован (OTP-поток — часть MVP).
- **Триггер:** Клиент ищет подходящее занятие.

### Основной поток
1. Открывает список слотов на 7 дней, сортировка по времени начала.
2. Заполненные и отменённые — с «Мест нет», CTA неактивна.
3. (Опц.) Задаёт фильтры.
4. Видит отфильтрованный список.

### Альтернативные потоки
- **A1. Сброс фильтров** → дефолтная выдача 7 дней.
- **A2. Только свободные** → скрыты заполненные/отменённые.
- **A3. Пустой справочник мастеров** → фильтр по мастеру скрыт.

### Исключения
- **E1. Нет слотов под фильтр** → empty state с подсказкой.
- **E2. Нет расписания вовсе** → «Пока нет доступных занятий».
- **E3. Офлайн** → кэш с пометкой «данные могли устареть».

### Постусловия
- Клиент видит актуальный (или помеченный) список; расписание не изменено.

## UC-4: Отмена занятия мастерской (форс-мажор)

- **Актёры:** Мастерская (инфраструктура), Клиент (получатель)
- **Предусловия:** Есть активные брони на слот; мастерская отменяет занятие (печь, электричество).
- **Триггер:** Форс-мажор на стороне мастерской.

### Основной поток (клиентская часть, R-008)
1. Слот получает статус `cancelled` в API.
2. Брони клиентов переходят в `workshop_cancelled` с **текстовой причиной**.
3. **Must:** в приложении — статус «Отменено мастерской» и причина; CTA записи неактивна.
4. **Should:** push-уведомление (в mock — симуляция / локальное уведомление).

### Постусловия
- Бронь сохранена в истории с причиной; повторная запись на слот запрещена.

## UC-5: Авторизация клиента (OTP)

- **Актёры:** Клиент
- **Предусловия:** Приложение установлено.
- **Триггер:** Первый запуск или сессия истекла.

### Основной поток
1. Клиент вводит номер телефона.
2. Получает SMS-код (OTP), вводит код.
3. Если новый пользователь — вводит имя.
4. Система выдаёт токен сессии; клиент попадает на список слотов.

### Исключения
- **E1. Неверный код** — повтор ввода.
- **E2. Rate limit** — сообщение «слишком много попыток».

### Постусловия
- Клиент авторизован; доступны слоты и бронирования.
